## Slack や O365 の API は機密情報をヘッダーで指定する。パスやクエリでないのは何故かセキュリティの観点で記載しなさい。

1. パスやクエリに含めるとログやキャッシュに残りやすい

   URL は以下の例のように多くのシステムで記録され、セキュリティ対策が甘い場合は第三者がアクセスできる可能性がある。

- Web サーバーのアクセスログ（Apache, Nginx など）
- ロードバランサーや API Gateway のログ
- ブラウザの履歴やブックマーク
- ネットワーク監視ツール

2. キャッシュやブックマークのリスク回避

   URLはブラウザやプロキシでキャッシュされることがある。

   機密情報がクエリに含まれていると、キャッシュファイルやブックマークに残り、漏洩リスクが高まる。

3. HTTPS による暗号化の適用範囲

   HTTPS は ヘッダーとボディを暗号化するが、URL の一部（ホスト名やパス）は 見える場合がある。

4. HTTP 標準のセキュリティ慣習

   RFC 7235 では、認証情報は Authorization ヘッダーで送信することが標準化されている。

   OAuth 2.0 の Bearer トークンも Authorization: Bearer <token> で送信するのが仕様。

   これは、HTTP の設計思想として「認証情報はリソース識別子（URL）とは分離すべき」という考え方に基づいる。
