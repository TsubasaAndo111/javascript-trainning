# 清一色 待ち当てタイムアタック

麻雀の清一色（マンズのみ）の手牌を見て、待ち牌を当てる60秒タイムアタックゲームです。

---

## 使い方

### 起動方法

`index.html` をブラウザで開く（ローカルサーバー経由推奨）。

### 画面構成

```
タイトル画面
├── ゲームスタート → ゲーム画面
└── ランキングを見る → ランキング画面

ゲーム画面
└── タイムアップ後、自動でランキング画面へ

ランキング画面
└── タイトルへ戻る → タイトル画面
```

### ゲームの流れ

1. 「ゲームスタート」を押すと60秒のカウントダウン開始
2. マンズ13枚の手牌が画像で表示される
3. 待ち牌と思う牌のボタンを選択（複数選択可）
   - ノーテン（聴牌していない）と思う場合は「ノーテン」ボタンを選択
   - 待ち牌ボタンを選ぶと「ノーテン」選択が解除され、逆も同様
4. 「回答する」ボタンで採点
   - 正解：**+10点**
   - 不正解：**−5点**（正解の待ち牌も表示）
5. 0.8秒後に次の問題へ自動遷移
6. 60秒経過でゲーム終了 → スコアが保存されランキング画面へ

### ランキング画面

- **歴代自己ベスト TOP3**：過去のプレイ結果からスコア上位3件を表示
- **成長曲線グラフ**：日付ごとの平均スコアを折れ線グラフで表示（Chart.js使用）

> スコアはブラウザの `localStorage` に保存されます。ブラウザのデータを削除すると記録も消えます。

## ロジック解説

### モジュール依存関係

```
main.js（司令塔）
├── clock.js     時計表示
├── ui.js        画面制御・ゲーム進行
│   ├── logic.js   問題生成・アガリ判定
│   └── storage.js スコア保存・ランキング
```

---

### logic.js — コアロジック

#### 問題生成 (`getQuestion`)

50%の確率で2つのルートに分岐する。

- **聴牌確定ルート**：有効なアガリ形14枚を生成 → ランダムに1枚抜いて13枚に → 待ちが存在することを確認してから採用
- **完全ランダムルート**：1〜9を4枚ずつ計36枚のデッキをシャッフルして13枚抽出（高確率でノーテン）

#### アガリ形生成 (`generateAgariHand`)

- 雀頭を1種ランダムに選び2枚配置
- 面子を4つ生成（30%の確率で刻子、70%で順子）
- 同じ牌が5枚以上になったら失敗 → `getQuestion` がリトライ

#### 待ち牌計算 (`calculateWaits`)

- 1〜9を順番に手牌に1枚加えて14枚にし、`isAgari` でアガリ判定
- 既に4枚ある牌はスキップ（5枚目は存在しない）

#### アガリ判定 (`isAgari`)

まず手牌を枚数カウント配列に変換する。

```
手牌: [1, 1, 2, 3, 4, 4, 5, 6, 7, 7, 7, 8, 9, 9]
         ↓
counts: [0, 2, 1, 1, 2, 1, 1, 3, 1, 2]
        (インデックス0は未使用、1〜9が各牌の枚数)
```

判定は2段階で行う。

**判定1 — 七対子**：`counts[i] === 2` の牌を数えて7種あれば上がり。4枚持ちは `=== 2` にならないため対子としてカウントされない。

**判定2 — 4面子1雀頭**：`counts[i] >= 2` の牌を雀頭候補として順番に2枚仮抜きし、残り12枚を `checkMentsu` で面子分解できれば上がり。

#### `checkMentsu` — バックトラック再帰

**基本戦略**：常に「一番小さい番号の牌」から処理する。これにより探索を一方向に限定できる。

```
checkMentsu(counts)
├── 残り枚数が0 → true（全て面子に分解できた）
├── 最小番号の牌 i を探す
├── パターンA: counts[i] >= 3 なら刻子として3枚消費 → 再帰
│     失敗したら3枚を元に戻す（バックトラック）
└── パターンB: counts[i,i+1,i+2] が全て >= 1 なら順子として消費 → 再帰
      失敗したら3枚を元に戻す（バックトラック）
      ※ i >= 8 の場合は i+2 が存在しないため試みない
```

**バックトラック例**：

```
counts = [0, 0,0,2,1,0,0,0,0,0]  ← 3が2枚、4が1枚（i=3）

パターンA: counts[3] >= 3? → NO（2枚しかない）
パターンB: counts[3,4,5] >= 1? → counts[5]=0 → NO
→ どちらも失敗 → false（この雀頭の仮定を棄却）
```

**複雑な探索例**：

```
checkMentsu([0, 3,1,1,1,1,1,1,1,2])  ← 9の雀頭2枚抜き後

i=1, counts[1]=3
パターンA(刻子): 1,1,1 消費
  checkMentsu([0, 0,1,1,1,1,1,1,1,2])
    i=2
    パターンB(順子): 2,3,4 消費
      checkMentsu([0, 0,0,0,0,1,1,1,1,2])
        i=5
        パターンB: 5,6,7 消費
          checkMentsu([0, 0,0,0,0,0,0,0,1,2])
            i=8
            パターンA: NO / パターンB: counts[10]なし → NO
            → false ❌ バックトラック
    → false ❌ バックトラック
→ パターンB(順子): 1,2,3 消費
  checkMentsu([0, 2,0,0,1,1,1,1,1,2])
    ...（探索継続）
```

深さ優先で全パターンを試し、一つでも成功すれば `true` を返す。

#### 設計の核心

| 特徴 | 内容 |
|---|---|
| 最小番号優先 | 常に一番小さい牌から処理することで探索を一方向に限定 |
| バックトラック | 失敗したら枚数を元に戻して別パターンを試す |
| 再帰の終了 | 全枚消費できたら成功（ベースケース） |
| 雀頭との責務分離 | `isAgari` が雀頭の仮定を担い、`checkMentsu` は面子のみを判定 |

---

### ui.js — ゲーム進行

| フェーズ | 処理 |
|---|---|
| ゲーム開始 | スコア0・残り60秒でリセット、1秒ごとにカウントダウン |
| 問題表示 | 手牌を画像で表示、1〜9の待ち牌選択ボタンを生成 |
| 回答 | 正解+10点 / 不正解-5点、0.8秒後に次問へ |
| 時間切れ | スコア保存、5秒後にランキング画面へ遷移 |

---

### storage.js — データ永続化

- `localStorage` のキー `mahjong_chinitsu_history` にプレイ履歴（日付+スコア）をJSON配列で保存
- `getRankingTop3`：スコア降順でTOP3を取得
- `getDailyAverages`：日付ごとにスコアを集計し平均を算出 → Chart.js で折れ線グラフに描画
